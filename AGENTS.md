
LLM 코딩 에이전트가 이 프로젝트에서 **필요한 부분만 정확히 바꾸고, 책임을 섞지 않으며, 검증 가능한 결과**를 내기 위한 지침입니다.

목표는 코드를 무조건 작게 쪼개는 것이 아닙니다.  
목표는 **변경 이유가 분명하고, 읽기 쉽고, 테스트하기 쉬운 코드**를 만드는 것입니다.

이 지침은 속도보다 신중함을 우선합니다.  
다만 단순한 오타 수정, 명백한 한 줄 변경, 충분히 정보가 주어진 작업에서는 사소한 확인 질문 때문에 멈추지 마세요. 필요한 가정을 짧게 말하고 진행하세요.

디자인 관련 작업을 할 때는 `docs/DESIGN_GUIDELINES.md`를 먼저 확인하고, 그 기준을 따라 화면 통일감을 유지하세요.

## 0. 우선순위

지침이 충돌하면 다음 순서로 따르세요.

1. 현재 사용자의 명시적 요청
2. 현재 작업 위치에 더 가까운 `AGENTS.md`
3. 이 문서의 공통 지침
4. 기존 코드베이스의 스타일과 구조

보안 지침은 항상 우선합니다.  
비밀키, API 키, 토큰, 비밀번호는 코드, 로그, 테스트 데이터, 예시값에 직접 넣지 마세요.

## 1. 작업 전에 목표를 분명히 하기

구현을 시작하기 전에 사용자 요청을 검증 가능한 목표로 바꾸세요.

- 해석이 여러 가지이고 결과가 달라진다면 먼저 질문하세요.
- 해석이 여러 가지 가능하다면 조용히 하나를 고르지 말고 가능한 해석을 짧게 제시하세요.
- 정보가 충분하다면 가정을 짧게 밝히고 진행하세요.
- 더 단순하거나 안전한 방법이 있다면 제안하세요.
- 사용자 요청이 기존 구조를 해칠 수 있다면 이유와 대안을 설명하세요.
- 이해하지 못한 부분을 아는 척하지 마세요.

질문이 필요한 경우:

- 어떤 파일, 화면, 기능을 바꿔야 하는지 특정할 수 없습니다.
- 성공 기준이 불명확해 테스트나 확인 방법을 정할 수 없습니다.
- 요청대로 진행하면 데이터 손실, 보안 문제, 큰 구조 변경이 생길 수 있습니다.
- 사용자의 의도에 따라 구현 방식이나 결과물이 크게 달라질 수 있습니다.

질문 없이 진행해도 되는 경우:

- 기존 코드와 테스트를 보면 의도가 분명합니다.
- 작은 수정이며 되돌리기 쉽습니다.
- 합리적인 기본값을 둘 수 있고 그 가정을 설명할 수 있습니다.
- 사용자가 이미 충분한 맥락과 성공 기준을 제공했습니다.

## 2. 단순하고 필요한 변경만 하기

문제를 해결하는 최소한의 코드를 작성하세요.

하지 마세요:

- 요청받지 않은 기능 추가
- 한 번만 쓰이는 코드를 위한 추상화
- 불필요한 설정 가능성, 확장성, 유연성 추가
- 실제로 일어나기 어려운 상황을 위한 과한 에러 처리
- 요청과 무관한 리팩터링, 포맷 변경, 주석 정리
- 고장 나지 않은 코드를 취향 때문에 바꾸기

지키세요:

- 기존 코드 스타일을 따릅니다.
- 모든 변경 줄은 사용자 요청과 직접 연결되어야 합니다.
- 내가 추가한 코드 때문에 불필요해진 import, 변수, 함수는 제거합니다.
- 기존부터 있던 죽은 코드나 미사용 코드는 언급만 하고 삭제하지 않습니다.
- 더 짧고 명확한 구현이 가능하다면 다시 단순하게 만듭니다.

단순함은 모든 코드를 한 파일에 몰아넣는다는 뜻이 아닙니다.  
책임이 섞여 읽기 어려워진다면 작은 분리가 더 단순한 해결책일 수 있습니다.

스스로 확인하세요.

- 숙련된 개발자가 이 코드를 보고 너무 복잡하다고 느낄까?
- 이 코드를 이해하려고 관련 없는 내용까지 읽어야 할까?
- 더 짧고 명확한 구현이 가능한가?
- 지금 추가한 코드가 사용자 요청과 직접 연결되는가?

## 3. 필요한 부분만 정확히 수정하기

반드시 필요한 부분만 건드리세요.  
내가 만든 변경 때문에 생긴 문제만 정리하세요.

기존 코드를 수정할 때는 다음을 지키세요.

- 주변 코드, 주석, 포맷을 괜히 “개선”하지 마세요.
- 고장 나지 않은 코드를 요청과 무관하게 리팩터링하지 마세요.
- 내가 더 좋은 방식이 있다고 생각해도 기존 코드 스타일을 따르세요.
- 관련 없는 죽은 코드나 미사용 코드를 발견하면 언급만 하세요. 요청받지 않았다면 삭제하지 마세요.

단, 요청을 안전하게 구현하기 위해 필요한 작은 정리는 허용됩니다.

- 새 코드 때문에 책임이 섞이는 것을 막기 위한 작은 함수 분리
- 새 기능을 테스트하기 쉽게 만드는 핵심 로직 분리
- 내가 추가한 코드 때문에 불필요해진 import, 변수, 함수 제거
- 변경 범위 안에서만 필요한 이름 정리

검증 기준:

- 모든 변경 줄은 사용자의 요청과 직접 연결되어 있어야 합니다.
- 관련 없는 정리는 따로 언급만 하고, 요청받지 않았다면 실행하지 마세요.
- 작업 도중 발견한 별도 문제는 이번 변경에 섞지 말고 별도 제안으로 남기세요.

## 4. 책임 경계를 지키기

최소 변경은 “기존 파일에 무조건 덧붙이라”는 뜻이 아닙니다.  
최소 변경은 **요청을 해결하면서 책임 경계를 망가뜨리지 않는 가장 작은 변경**입니다.

서로 섞지 말아야 할 책임:

- 화면 표시
- 사용자 입력 처리
- 핵심 규칙과 계산
- 데이터 저장과 불러오기
- 외부 API, 파일, 네트워크 연결

기존 구조가 있다면 먼저 따르세요.  
새로운 구조를 만들기 전에 기존 위치 중 어디가 가장 자연스러운지 확인하세요.

다만 기존 구조가 이미 여러 책임을 섞고 있다면, 그대로 따라가기 전에 작은 책임 분리를 제안하세요.  
큰 구조 변경이 필요하다면 바로 실행하지 말고, 먼저 이유와 범위를 설명하세요.

같은 파일에 둬도 되는 코드:

- 그 파일에서만 쓰이고 가까이 있을 때 더 읽기 쉬운 작은 보조 코드
- 간단한 포맷팅, 라벨 생성, 표시용 변환
- 특정 화면이나 기능의 표현만 돕는 코드
- 분리했을 때 파일 이동만 늘고 의미가 흐려지는 코드

분리를 검토해야 하는 코드:

- 여러 곳에서 다시 쓰일 수 있는 코드
- 테스트하고 싶은 중요한 규칙이나 계산
- 외부 입출력 없이도 의미가 있는 핵심 로직
- 여러 책임이 섞여 변경하기 어려운 코드
- UI 코드 안에 묻힌 입력 검증, 권한 판단, 가격 계산, 상태 변경 규칙
- API 호출 코드 안에 섞인 데이터 변환이나 비즈니스 규칙

코드를 옮길 때는 위치만 바꾸지 마세요.  
분리 후 역할이 더 분명해져야 합니다.

스스로 확인하세요.

- 이 파일을 한 문장으로 설명할 수 있는가?
- 이 함수는 하나의 작은 행동만 하는가?
- 이 코드는 여기 있을 때 더 읽기 쉬운가?
- 이 코드는 무엇이 바뀔 때 함께 바뀌어야 하는가?

## 5. 파일 비대화 막기

파일이 큰 것 자체가 항상 문제는 아닙니다.  
문제는 한 파일에 서로 다른 변경 이유가 계속 쌓이는 것입니다.

새 코드를 기존 파일에 넣기 전에 확인하세요.

- 새 코드가 그 파일의 한 문장 설명에 들어맞는가?
- 화면, 입력 처리, 핵심 규칙, 저장, 외부 API 중 여러 책임이 섞이는가?
- 테스트하고 싶은 핵심 로직이 UI나 네트워크 코드 안에 묻히는가?
- 같은 파일을 서로 다른 이유로 자주 고치게 될 가능성이 높은가?
- 새 기능 때문에 파일을 이해하기 위해 읽어야 하는 범위가 크게 늘어나는가?
- “일단 여기 넣자”는 임시 코드가 반복되고 있는가?

분리가 필요해 보이면 다음 순서로 판단하세요.

1. 기존 폴더 구조 안에 자연스러운 위치가 있는지 찾습니다.
2. 새 파일이 필요하다면 그 파일의 책임을 한 문장으로 설명합니다.
3. 분리 전후로 동작이 같다는 것을 테스트나 실행 결과로 확인합니다.
4. 분리 때문에 import, 이름, 경로만 복잡해진다면 다시 합칩니다.

작은 분리는 과한 설계가 아닙니다.  
책임이 섞이는 것을 막기 위한 가장 작은 변경일 수 있습니다.

## 6. 새 라이브러리는 신중하게 추가하기

새 라이브러리는 금지하지 않습니다.  
다만 몇 줄의 명확한 코드로 충분하거나 이미 같은 역할의 도구가 있다면 추가하지 마세요.

추가 전에 확인하세요.

- 표준 라이브러리나 기존 의존성으로 충분한가?
- 직접 구현하면 더 위험한 영역인가?
- 런타임 의존성인가, 개발 도구인가?
- 유지보수 상태, 보안 이슈, 라이선스, 번들 크기는 괜찮은가?
- 간접 의존성이 과하게 늘어나지 않는가?
- 프로젝트의 기존 기술 선택과 잘 맞는가?

직접 구현보다 검증된 라이브러리가 더 안전할 수 있는 영역:

- 인증, 권한, 암호화, 보안 관련 처리
- 날짜, 시간, 타임존 처리
- 복잡한 입력 검증이나 파싱
- 접근성, 국제화, 인코딩 처리
- 파일 포맷, 네트워크 프로토콜, 결제처럼 실수 비용이 큰 영역

새 라이브러리를 추가하지 말아야 하는 경우:

- 몇 줄의 명확한 코드로 충분합니다.
- 이미 프로젝트 안에 같은 역할의 도구가 있습니다.
- 한 번만 쓰는 작은 편의 기능입니다.
- 라이브러리 사용법을 이해하지 못한 상태입니다.
- 문제를 단순화하는 대신 프로젝트 전체 복잡도를 늘립니다.

라이브러리가 필요하다고 판단했다면 추가 전 짧게 설명하세요.

```md
왜 필요한가:
대안:
선택한 이유:
추가되는 비용:
검증 방법:
```

## 7. 목표 중심으로 실행하고 검증하기

작업 지시를 검증 가능한 목표로 바꾸세요.

예시:

- “검증 로직 추가해줘” → 잘못된 입력 테스트를 먼저 작성하고 통과시키기
- “버그 고쳐줘” → 버그를 재현하는 테스트를 먼저 작성하고 통과시키기
- “리팩터링해줘” → 리팩터링 전후로 기존 테스트가 통과하는지 확인하기

가능하면 자동 테스트로 검증하세요.  
테스트를 실행하지 못했다면 이유를 솔직히 말하고, 사용자가 직접 확인할 수 있는 방법을 제시하세요.

검증할 때 확인하세요.

- 변경 전 문제를 재현했는가?
- 변경 후 같은 문제가 해결됐는가?
- 기존 기능이 깨지지 않았는가?
- 테스트, 린트, 타입 체크, 빌드 중 어떤 것을 실행했는가?
- 실행하지 못한 검증이 있다면 이유와 대체 확인 방법을 설명했는가?

## 8. 사용자에게 설명하는 방식

사용자는 바이브코딩 입문자이며 코드를 읽는 데 익숙하지 않습니다.

항상 지키세요.

- 설명은 한국어로 합니다.
- 어려운 개발 용어는 짧게 풀어서 설명합니다.
- 코드를 바꾸기 전에는 무엇을 바꿀지 요약합니다.
- 변경 후에는 어떤 파일을 왜 바꿨는지 쉽게 설명합니다.
- 실행 명령어는 복사해서 붙여넣을 수 있게 제공합니다.
- 테스트 결과를 말합니다.
- 테스트를 못 했다면 이유와 대체 확인 방법을 말합니다.
- 사용자가 직접 확인할 수 있는 체크리스트를 제공합니다.
- 큰 구조 변경, 파일 삭제, 새 라이브러리 추가는 꼭 필요할 때만 하고 이유를 설명합니다.
- 파일 분리나 새 파일 생성이 필요하다면 “왜 기존 파일에 넣지 않는지” 설명합니다.
- 새 라이브러리 추가가 필요하다면 “왜 직접 구현하지 않는지” 설명합니다.

불필요하게 장황하게 설명하지 마세요.  
사용자가 바로 다음 행동을 할 수 있을 만큼만 구체적으로 설명하세요.

## 9. 응답 형식

복잡한 작업 전:

```md
제가 이해한 목표:
바꿀 가능성이 있는 파일:
진행 방법:
확인 방법:
필요한 가정:
```

작업 후:

```md
변경한 파일:
- [파일명]: [무엇을 왜 바꿨는지]

검증:
- [실행한 테스트 또는 명령어]: [결과]

확인 체크리스트:
- [ ] [사용자가 확인할 항목]
- [ ] [사용자가 확인할 항목]

참고:
- [테스트를 못 했거나 남은 위험이 있다면 적기]
```

단순한 오타 수정, 명백한 한 줄 변경, 설명만 필요한 답변에서는 작업 전 형식을 생략해도 됩니다.

## 10. 잘 작동하는 상태의 기준

이 지침이 잘 작동하고 있다면 다음 결과가 보여야 합니다.

- diff에 불필요한 변경이 줄어듭니다.
- PR이나 변경사항이 작고 명확합니다.
- 구현 전 애매한 점을 숨기지 않습니다.
- 사소한 질문 때문에 작업이 불필요하게 멈추지 않습니다.
- 파일이 커질 때 책임 분리 여부를 먼저 검토합니다.
- 새 라이브러리를 추가할 때 이유와 대안을 함께 설명합니다.
- 테스트 또는 수동 확인 방법이 항상 제시됩니다.
- 사용자가 다음에 무엇을 확인해야 하는지 쉽게 알 수 있습니다.

## 테스트 완료 후 실행 파일 준비

WPF 앱 변경 작업을 마치고 테스트까지 성공했다면, 사용자가 레포 루트의 실행 파일을 더블클릭해서 바로 실험할 수 있도록 Windows x64 단일 파일 exe를 준비하세요.

기본 흐름은 다음과 같습니다.

1. 필요한 테스트를 실행하고 성공 여부를 확인합니다.
2. 테스트가 성공하면 아래 명령으로 최신 코드 기준 단일 파일 exe를 만듭니다.

```powershell
.\scripts\publish-win-x64.cmd -SelfContained -SingleFile
```

3. 생성된 실행 파일을 레포 루트의 `DeckDeckDeck.exe`로 복사합니다.
   복사 전에 현재 실행 중인 `DeckDeckDeck.exe` 프로세스가 있다면, 파일 잠금을 풀기 위해 종료해도 됩니다.

```powershell
Get-Process -Name DeckDeckDeck -ErrorAction SilentlyContinue | Stop-Process
```

```powershell
Copy-Item .\artifacts\publish\win-x64\DeckDeckDeck.App.exe .\DeckDeckDeck.exe -Force
```

4. 복사까지 성공했다면 실행 파일을 한 번 실행해 사용자가 바로 확인할 수 있게 합니다.

```powershell
.\DeckDeckDeck.exe
```

5. 최종 답변에서 `DeckDeckDeck.exe`를 더블클릭하면 된다고 알려주고, 테스트 결과, exe 생성 여부, 실행 시도 여부를 함께 설명합니다.

주의할 점:

- 테스트가 실패했다면 실패 이유를 먼저 설명하고, `DeckDeckDeck.exe`를 최신 성공 빌드처럼 말하지 마세요.
- 실행에 실패했다면 실패 이유를 설명하세요.
- `DeckDeckDeck.exe`는 로컬 실행용 결과물이므로 Git에 커밋하지 마세요.
