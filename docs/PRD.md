# DeckDeckDeck

## 0. 문서 목적

이 문서는 DeckDeckDeck의 현재 구현 상태를 기준으로 다음 MVP 범위를 다시 정리한다.

기존 PRD v0.3에는 이미 구현된 기능과 다음 MVP 후보가 섞여 있었다.  
따라서 v0.4에서는 현재 구현된 기능을 완료 상태로 올리고, 다음 MVP에서는 아직 없는 데이터 안전 기능만 작게 다룬다.

시각 디자인 기준은 계속 [DESIGN_GUIDELINES.md](DESIGN_GUIDELINES.md)를 따른다.

## 1. 제품 정의

DeckDeckDeck은 Windows에서 텐키 단축키와 텐키 모양 UI를 사용해 저장된 실행 항목을 빠르게 실행하는 개인용 로컬 데스크톱 앱이다.

현재 카드는 아래 세 가지 실행 결과를 가질 수 있다.

1. 문구 붙여넣기
2. 프로그램 또는 파일 실행
3. 웹페이지 주소 열기

핵심 사용 흐름은 다음과 같다.

```text
Ctrl + Numpad 0
→ 홈 화면
→ 덱 선택
→ 카드 선택
→ 문구 붙여넣기, 프로그램/파일 실행, 또는 웹페이지 열기
```

또는:

```text
Ctrl + Numpad 1~9 또는 기호
→ 해당 덱 바로 진입
→ 카드 선택
→ 문구 붙여넣기, 프로그램/파일 실행, 또는 웹페이지 열기
```

## 2. 현재 완료된 MVP

아래 항목은 현재 앱에서 완료된 기준 기능으로 본다.

| 영역 | 완료 내용 |
|---|---|
| 기술 기반 | C# / .NET 10 / WPF / MVVM / SQLite 기반 앱 구조 |
| 로컬 저장 | `%AppData%` 아래 DB, 이미지, 썸네일, 자동 아이콘 캐시, 로그 저장 |
| 홈 화면 | 텐키 모양의 덱 선택 화면 |
| 덱 화면 | 텐키 모양의 카드 선택 화면 |
| 카드 키 | `0~9`, `/`, `*`, `-`, `+`, `.` 15개 슬롯 |
| 전역 단축키 | `Ctrl + Numpad 0`, `Ctrl + Numpad 1~9`, `Ctrl + Numpad 기호` |
| 덱 관리 | 덱 추가, 수정, 삭제 |
| 카드 관리 | 실행 항목 추가, 수정, 삭제 |
| 긴 텍스트 저장 | 줄바꿈, Markdown, 코드블록 원문 저장 |
| 이미지 | 이미지 선택, 앱 내부 복사 저장, 썸네일 표시, 이미지 제거 |
| 자동 아이콘 | 프로그램/파일 실행 카드의 자동 아이콘 캐시 저장 및 표시 |
| 카드 편집 | 카드 우측 상단 톱니모양 아이콘으로 편집 진입 |
| 빠른 편집 | 활성 카드와 빈 카드는 카드 우클릭으로 편집 진입 |
| 슬롯 활성화 | 덱 슬롯과 카드 슬롯의 활성화/비활성화 설정 |
| 문구 붙여넣기 | 클립보드 임시 교체 후 `SendInput`으로 `Ctrl+V` 실행 |
| 클립보드 복원 | 붙여넣기 후 기존 클립보드 복원 |
| 프로그램/파일 실행 | `.exe`, 일반 파일, 폴더를 Windows 기본 방식으로 실행 |
| 웹페이지 주소 열기 | `http://` 또는 `https://` 주소를 Windows 기본 브라우저로 열기 |
| 웹 주소 자동 보정 | 사용자가 프로토콜 없이 입력한 주소에 `https://`를 붙여 저장 |
| 실행 실패 처리 | 실행 대상이 없거나 주소가 잘못되어도 앱이 종료되지 않고 상태 메시지와 로그를 남김 |
| 자동 숨김 | 문구 붙여넣기, 프로그램/파일 실행, 웹페이지 열기 후 창 자동 숨김 설정 적용 |
| 자동 백업 설정 | 설정 화면에서 자동 백업 사용 여부와 외부 백업 폴더 지정 |
| 변경 후 자동 백업 | 덱/카드/설정 변경 성공 후 백업 ZIP 자동 생성 |
| 수동 백업 | 설정 화면에서 즉시 백업 실행 |
| 백업 보관 | 자동 백업은 최근 10개를 기본으로 유지하고 오래된 자동 백업 정리 |
| 백업 실패 처리 | 백업 실패 시 앱이 종료되지 않고 상태 메시지와 로그를 남김 |
| 설정 화면 | 자동 숨김, 클립보드 복원, 자동 백업, 단축키 안내, 권한 안내 |
| 관리자 권한 안내 | 관리자 권한 앱/보호된 입력창에서 제한될 수 있음을 안내 |
| 트레이 | 트레이 아이콘에서 열기/종료 가능 |
| 창 위치 | 마지막 창 위치 저장 및 복원 |
| 단일 실행 안정성 | 사용자가 여러 번 실제 사용하며 안정적으로 작동함을 확인 |
| 실제 붙여넣기 안정성 | 사용자가 여러 앱에서 반복 사용하며 안정적으로 작동함을 확인 |

## 4. 다음 MVP 목표

다음 MVP의 목표는 **데이터 안전성 보강**이다.

현재 앱은 백업 ZIP을 만들 수 있지만, 다음 기능은 아직 없다.

1. DB 오류 시 앱이 반복 크래시하지 않게 막는 최소 안전장치
2. 기존 백업 ZIP을 선택해 현재 앱 데이터로 복원하는 기능

이번 MVP에서는 새 실행 타입이나 새 UI 기능을 넓히지 않는다.  
사용자가 이미 만든 덱, 카드, 문구, 실행 경로, 웹 주소, 이미지 데이터를 잃지 않도록 안전장치를 보강하는 데 집중한다.

## 5. 다음 MVP 포함 범위

### 5.1 DB 오류 대응

DB 오류 대응은 앱의 저장 파일이 잠기거나 손상되었을 때 앱이 시작하자마자 계속 꺼지는 상황을 줄이는 기능이다.

예를 들어 다음 상황을 다룬다.

- 앱이 저장하는 중 컴퓨터가 꺼져 DB 파일이 손상된 경우
- 백신, 동기화 프로그램, 다른 프로세스가 DB 파일을 잠깐 잡고 있는 경우
- DB 파일 경로가 이상하거나 접근 권한 문제가 생긴 경우

| ID | 요구사항 |
|---|---|
| FR-DB-001 | DB를 열거나 초기화하는 중 오류가 나도 앱이 반복 크래시하지 않아야 한다 |
| FR-DB-002 | DB 오류가 발생하면 사용자가 이해할 수 있는 상태 메시지 또는 안내를 보여준다 |
| FR-DB-003 | DB 오류 상세 내용은 로그에 남긴다 |
| FR-DB-004 | DB 오류 대응 중 기존 DB 파일을 임의로 삭제하지 않는다 |
| FR-DB-005 | 손상된 DB를 새 DB로 교체해야 하는 경우, 교체 전에 가능한 한 기존 DB 파일을 별도 백업한다 |
| FR-DB-006 | DB 오류 상태에서도 앱이 최소한 종료 안내 또는 복구 안내를 보여줄 수 있어야 한다 |

주의:

- 이번 MVP에서 복잡한 자동 복구까지 만들 필요는 없다.
- 사용자가 만든 데이터를 조용히 삭제하거나 덮어쓰면 안 된다.
- DB 오류 대응은 백업 복원 기능과 연결될 수 있지만, 두 기능은 분리해서 구현한다.

성공 기준:

- DB 파일 잠금 또는 잘못된 DB 경로 상황에서 앱이 반복 크래시하지 않는다.
- DB 오류가 발생하면 사용자가 원인을 대략 이해할 수 있는 안내를 본다.
- 오류 내용이 로그에 남는다.
- 기존 DB 파일이 사용자 동의 없이 삭제되지 않는다.

### 5.2 백업 ZIP 복원

현재 앱은 백업 ZIP을 만들 수 있지만, 백업 ZIP을 앱 안에서 복원하는 기능은 없다.  
다음 MVP에서는 사용자가 백업 ZIP을 선택해 현재 앱 데이터로 되돌릴 수 있게 한다.

| ID | 요구사항 |
|---|---|
| FR-RESTORE-001 | 사용자는 설정 화면에서 백업 ZIP 복원을 시작할 수 있다 |
| FR-RESTORE-002 | 사용자는 기존 DeckDeckDeck 백업 ZIP 파일을 선택할 수 있다 |
| FR-RESTORE-003 | 앱은 선택한 ZIP 안에 필수 파일인 `launcher.db`가 있는지 확인한다 |
| FR-RESTORE-004 | 앱은 ZIP 안의 `images`, `icon-cache` 폴더가 있으면 함께 복원한다 |
| FR-RESTORE-005 | 복원 전에 현재 앱 데이터를 안전 백업으로 한 번 더 저장한다 |
| FR-RESTORE-006 | 복원은 현재 DB, 이미지, 썸네일, 자동 아이콘 캐시를 백업 ZIP 내용으로 교체한다 |
| FR-RESTORE-007 | 복원 실패 시 가능한 한 기존 현재 데이터가 유지되어야 한다 |
| FR-RESTORE-008 | 복원이 끝나면 앱 재시작이 필요하다는 안내를 보여준다 |
| FR-RESTORE-009 | 잘못된 ZIP, 필수 파일이 없는 ZIP, 읽을 수 없는 ZIP은 복원하지 않고 오류 메시지를 보여준다 |

복원 대상 범위:

- SQLite DB 파일 `launcher.db`
- 사용자가 선택해 앱 내부에 저장된 원본 이미지
- 이미지 썸네일
- 실행 파일/바로 가기 자동 아이콘 캐시
- DB 안에 저장된 설정값, 실행 경로, 웹 주소

복원에서 제외하는 범위:

- 사용자가 카드에 연결한 외부 실행 파일 자체
- 사용자가 카드에 연결한 외부 문서 파일 자체
- 앱 로그 파일
- 빌드 결과물, 실행 파일, 개발용 임시 파일

주의:

- 복원은 현재 데이터를 덮어쓰는 위험한 작업이다.
- 복원 전 안전 백업이 실패하면 복원을 진행하지 않는 방향을 우선 검토한다.
- 복원 중 앱이 DB를 사용하고 있으면 파일 교체가 실패할 수 있으므로, 복원 흐름은 앱 재시작 또는 안전한 교체 절차를 고려해야 한다.
- 백업 ZIP이 다른 버전에서 만들어졌더라도 현재 앱이 읽을 수 있는지 확인해야 한다.

성공 기준:

- 사용자가 백업 ZIP을 선택할 수 있다.
- 올바른 백업 ZIP을 선택하면 DB, 이미지, 썸네일, 자동 아이콘 캐시가 복원된다.
- 복원 전에 현재 데이터 안전 백업이 만들어진다.
- 잘못된 ZIP을 선택해도 앱이 종료되지 않는다.
- 복원 후 앱을 다시 열면 백업 시점의 덱과 카드가 표시된다.

## 7. 현재 데이터 구조

현재 `Snippet`은 문구뿐 아니라 실행 항목을 함께 저장한다.  
다만 사용자-facing 표현에서는 계속 카드 또는 실행 항목이라고 부른다.

| 필드 | 설명 |
|---|---|
| `ActionType` | `PasteText`, `LaunchFile`, `LaunchUrl` |
| `Content` | `PasteText`일 때 붙여넣을 문구 |
| `LaunchPath` | `LaunchFile`일 때 실행할 파일/폴더 경로 |
| `LaunchUrl` | `LaunchUrl`일 때 열 웹페이지 주소 |

현재 규칙:

- 기존 문구 데이터는 모두 `PasteText`로 유지된다.
- `LaunchFile`에서는 `Content`를 실행에 사용하지 않는다.
- `PasteText`에서는 `Content`가 필요하고, `LaunchPath`와 `LaunchUrl`은 비어 있을 수 있다.
- `LaunchFile`에서는 `LaunchPath`가 필요하고, `Content`와 `LaunchUrl`은 비어 있을 수 있다.
- `LaunchUrl`에서는 `LaunchUrl`이 필요하고, `Content`와 `LaunchPath`는 비어 있을 수 있다.
- 실행 타입 확장은 문구 붙여넣기 안정성을 해치면 안 된다.

백업 설정:

| 설정 | 설명 |
|---|---|
| `AutoBackupEnabled` | 변경 후 자동 백업을 사용할지 여부 |
| `BackupFolderPath` | 사용자가 지정한 외부 백업 폴더 경로 |
| `AutoBackupRetentionCount` | 유지할 최근 자동 백업 개수, 기본값 10 |
| `LastBackupCreatedAt` | 마지막 백업 생성 시각 |

백업 ZIP 현재 구조:

| 항목 | 설명 |
|---|---|
| `launcher.db` | SQLite DB 스냅샷 |
| `images/` | 원본 이미지와 썸네일 |
| `icon-cache/` | 프로그램/파일 실행 카드의 자동 아이콘 캐시 |

주의:

- 백업 폴더 경로는 앱 데이터 폴더 내부가 아니어야 한다.
- 백업 실패 기록은 로그에 남기되, 백업 ZIP 안에는 로그 파일을 포함하지 않는다.
- 창 위치 저장은 설정에 저장되지만 자동 백업 생성 트리거로 보지 않는다.

## 8. 다음 MVP 우선순위

1. DB 오류 대응
   - 확인 방법: DB 파일 잠금/손상/잘못된 DB 경로 상황에서도 앱이 반복 크래시하지 않는지 확인

2. 백업 ZIP 복원 기본 흐름
   - 확인 방법: 백업 ZIP을 선택하고 `launcher.db`, `images`, `icon-cache`를 앱 데이터로 되돌릴 수 있는지 확인

3. 복원 전 안전 백업
   - 확인 방법: 복원 직전에 현재 앱 데이터가 별도 ZIP으로 백업되는지 확인

4. 잘못된 ZIP 방어
   - 확인 방법: 필수 파일이 없는 ZIP, 깨진 ZIP, 읽을 수 없는 ZIP을 선택해도 앱이 종료되지 않는지 확인

5. 회귀 테스트
   - 확인 방법: 기존 문구 붙여넣기, 프로그램/파일 실행, 웹페이지 열기, 자동 백업, 클립보드 복원, 자동 숨김이 그대로 작동하는지 확인

## 9. 검증 체크리스트

다음 MVP 완료 시 아래를 확인한다.

- `dotnet test`가 통과한다.
- DB 오류 상황에서 앱이 반복 크래시하지 않는다.
- DB 오류가 발생하면 사용자 안내와 로그가 남는다.
- DB 오류 대응 중 기존 DB 파일이 조용히 삭제되지 않는다.
- 설정 화면에서 백업 ZIP 복원을 시작할 수 있다.
- 올바른 백업 ZIP을 선택하면 DB, 이미지, 썸네일, 자동 아이콘 캐시가 복원된다.
- 복원 전에 현재 데이터 안전 백업이 만들어진다.
- 잘못된 ZIP을 선택해도 앱이 종료되지 않는다.
- 복원 후 앱을 다시 열면 백업 시점의 덱과 카드가 표시된다.
- 기존 문구 카드가 그대로 붙여넣기 된다.
- 기존 프로그램/파일 실행 카드가 실행된다.
- 기존 웹페이지 주소 열기 카드가 기본 브라우저로 열린다.
- 기존 자동 백업이 계속 동작한다.
- 테스트 통과 후 최신 `DeckDeckDeck.exe`를 레포 루트에 준비한다.

## 10. 용어

| 용어 | 뜻 |
|---|---|
| 홈 화면 | 덱을 고르는 첫 화면 |
| 덱 화면 | 특정 덱 안의 카드를 고르는 화면 |
| 카드 | 텐키 한 키에 대응되는 실행 단위 |
| 실행 항목 | 카드에 저장된 실제 동작 정보 |
| 문구 카드 | 선택하면 저장된 텍스트를 붙여넣는 카드 |
| 프로그램/파일 실행 카드 | 선택하면 프로그램, 파일, 폴더를 여는 카드 |
| 웹페이지 주소 열기 카드 | 선택하면 저장된 웹 주소를 기본 브라우저로 여는 카드 |
| 슬롯 | 카드가 놓이는 텐키 위치 |
| 빠른 편집 | 활성 카드 또는 빈 카드를 우클릭해 바로 편집하는 동작 |
| 비활성 카드 | 현재 사용할 수 없도록 꺼 둔 카드 |
| 자동 백업 | 데이터 변경이 성공한 뒤 앱이 백업 파일을 자동으로 만드는 동작 |
| 수동 백업 | 사용자가 설정 화면에서 직접 즉시 만드는 백업 |
| 백업 폴더 | 백업 ZIP 파일을 저장할 사용자가 지정한 외부 폴더 |
| 백업 ZIP | DB와 이미지 등 앱 데이터를 하나로 묶은 압축 백업 파일 |
| 복원 | 백업 ZIP 내용을 현재 앱 데이터로 되돌리는 동작 |
| 안전 백업 | 복원 전에 현재 데이터를 보호하기 위해 한 번 더 만드는 백업 |
| DB 오류 대응 | 저장 파일 문제로 앱이 반복 종료되지 않도록 막는 안전장치 |
| 붙여넣기 모드 | 기존 입력창 위치를 유지한 채 카드를 고르는 상태 |
| 편집 모드 | DeckDeckDeck 창이 포커스를 가져와 내용을 수정하는 상태 |
