# PRD v0.6

# DeckDeckDeck

## 0. 문서 목적

이 문서는 DeckDeckDeck의 현재 구현 상태, 유지할 제품 정책, 다음 MVP, 이후 확장 후보를 정리한다.

기존 PRD v0.4의 **다음 MVP: 데이터 안전성 보강**은 유지한다.
즉, DB 오류 대응과 백업 ZIP 복원 범위는 삭제하거나 축소하지 않는다.

데이터 안전성 MVP 이후에는 DeckDeckDeck을 단순 문구 런처가 아니라 **키보드 기반 Stream Deck**으로 확장한다.
카드 실행 타입은 한 번에 크게 바꾸지 않고, 빠른 체감 기능부터 자동화, 컨텍스트 인식, 외부 서비스 연동 순서로 점진적으로 넓힌다.

시각 디자인 기준은 계속 [DESIGN_GUIDELINES.md](deckdeckdeck/docs/DESIGN_GUIDELINES.md)를 따른다.

## 1. 현재 제품 정의

DeckDeckDeck은 Windows에서 텐키 단축키와 텐키 모양 UI를 사용해 저장된 실행 항목을 빠르게 실행하는 개인용 로컬 데스크톱 앱이다.

현재 카드는 아래 네 가지 실행 결과를 가질 수 있다.

1. 문구 붙여넣기
2. 프로그램 또는 파일 실행
3. 웹페이지 주소 열기
4. 음악/미디어 제어

핵심 사용 흐름은 다음과 같다.

```text
Ctrl + Numpad 0
→ 홈 화면
→ 덱 선택
→ 카드 선택
→ 문구 붙여넣기, 프로그램/파일 실행, 웹페이지 열기, 또는 음악/미디어 제어
```

또는:

```text
Ctrl + Numpad 1~9 또는 기호
→ 해당 덱 바로 진입
→ 카드 선택
→ 문구 붙여넣기, 프로그램/파일 실행, 웹페이지 열기, 또는 음악/미디어 제어
```

후속 제품 방향:

- DeckDeckDeck은 로컬 Windows 앱을 유지한다.
- 텐키 기반 15개 슬롯 구조를 유지한다.
- 서버, 계정, 클라우드 동기화 없이 개인용 런처로 유지한다.
- 후속 기능은 카드 실행 타입과 덱 전환 방식을 점진적으로 확장한다.
- 서비스 전용 제어는 처음부터 별도 기능으로 나누지 않고, 같은 카드 타입의 provider 확장으로 본다.

## 2. 현재 완료된 MVP

아래 항목은 현재 앱에서 완료된 기준 기능으로 본다.

| 영역 | 완료 내용 |
|---|---|
| 기술 기반 | C# / .NET 10 / WPF / MVVM / SQLite 기반 앱 구조 |
| 로컬 저장 | `%AppData%` 아래 DB, 이미지, 썸네일, 자동 아이콘 캐시, 로그 저장 |
| 홈 화면 | 텐키 모양의 덱 선택 화면 |
| 덱 화면 | 텐키 모양의 카드 선택 화면 |
| 카드 키 | `0~9`, `/`, `*`, `-`, `+`, `.` 15개 슬롯 |
| 전역 단축키 | `Ctrl + Numpad 0`, `Ctrl + Numpad 1~9`, `Ctrl + Numpad 기호` |
| 덱 관리 | 덱 추가, 수정, 삭제 |
| 카드 관리 | 실행 항목 추가, 수정, 삭제 |
| 긴 텍스트 저장 | 줄바꿈, Markdown, 코드블록 원문 저장 |
| 이미지 | 이미지 선택, 앱 내부 복사 저장, 썸네일 표시, 이미지 제거 |
| 자동 아이콘 | 프로그램/파일 실행 카드의 자동 아이콘 캐시 저장 및 표시 |
| 카드 편집 | 카드 우측 상단 톱니모양 아이콘으로 편집 진입 |
| 빠른 편집 | 활성 카드와 빈 카드는 카드 우클릭으로 편집 진입 |
| 슬롯 활성화 | 덱 슬롯과 카드 슬롯의 활성화/비활성화 설정 |
| 문구 붙여넣기 | 클립보드 임시 교체 후 `SendInput`으로 `Ctrl+V` 실행 |
| 클립보드 복원 | 붙여넣기 후 기존 클립보드 복원 |
| 프로그램/파일 실행 | `.exe`, 일반 파일, 폴더를 Windows 기본 방식으로 실행 |
| 웹페이지 주소 열기 | `http://` 또는 `https://` 주소를 Windows 기본 브라우저로 열기 |
| 웹 주소 자동 보정 | 사용자가 프로토콜 없이 입력한 주소에 `https://`를 붙여 저장 |
| 음악/미디어 제어 | Windows 범용 미디어 키 기반으로 재생/일시정지, 이전 곡, 다음 곡, 정지, 음소거, 볼륨 증가, 볼륨 감소 실행 |
| 미디어 기본 아이콘 | Lucide 기반 투명 배경 기본 아이콘을 앱 리소스로 포함하고, 사용자 지정 이미지가 있으면 사용자 이미지를 우선 표시 |
| 실행 실패 처리 | 실행 대상이 없거나 주소가 잘못되어도 앱이 종료되지 않고 상태 메시지와 로그를 남김 |
| 자동 숨김 | 문구 붙여넣기, 프로그램/파일 실행, 웹페이지 열기, 음악/미디어 제어 후 창 자동 숨김 설정 적용 |
| 자동 백업 설정 | 설정 화면에서 자동 백업 사용 여부와 외부 백업 폴더 지정 |
| 변경 후 자동 백업 | 덱/카드/설정 변경 성공 후 백업 ZIP 자동 생성 |
| 수동 백업 | 설정 화면에서 즉시 백업 실행 |
| 백업 보관 | 자동 백업은 최근 10개를 기본으로 유지하고 오래된 자동 백업 정리 |
| 백업 실패 처리 | 백업 실패 시 앱이 종료되지 않고 상태 메시지와 로그를 남김 |
| 설정 화면 | 자동 숨김, 클립보드 복원, 자동 백업, 단축키 안내, 권한 안내 |
| 관리자 권한 안내 | 관리자 권한 앱/보호된 입력창에서 제한될 수 있음을 안내 |
| 트레이 | 트레이 아이콘에서 열기/종료 가능 |
| 창 위치 | 마지막 창 위치 저장 및 복원 |
| 단일 실행 안정성 | 사용자가 여러 번 실제 사용하며 안정적으로 작동함을 확인 |
| 실제 붙여넣기 안정성 | 사용자가 여러 앱에서 반복 사용하며 안정적으로 작동함을 확인 |

## 3. 현재 유지할 정책

아래 항목은 현재 상태를 유지한다.

| 항목 | 정책 |
|---|---|
| Disabled UI | 현재 UI에 만족하므로 다음 MVP에서 변경하지 않는다 |
| 비활성 카드 편집 | 비활성 카드는 우클릭 편집이 아니라 톱니모양 아이콘으로 편집한다 |
| 텐키 슬롯 수 | 15개 슬롯 구조를 유지한다 |
| 기본 단축키 | 현재 단축키 체계를 유지한다 |
| 클립보드 방식 | 직접 타이핑이 아니라 클립보드 + `Ctrl+V` 방식을 유지한다 |
| 로컬 우선 | 서버, 계정, 클라우드 동기화 없이 로컬 앱으로 유지한다 |
| 배포 범위 | 설치 프로그램, 자동 업데이트, 스토어 등록은 계속 제외한다 |
| 디자인 방향 | 기존 기본 팔레트와 Soft Keycap Card 방향을 유지한다 |
| 웹 주소 보정 | `example.com`처럼 프로토콜이 없는 주소는 `https://example.com`으로 보정한다 |
| 프로그램/파일 실행 명칭 | 사용자-facing 표현은 `프로그램/파일 실행`을 유지한다 |
| 창 위치 저장 | 창 위치 저장은 자동 백업을 만들지 않는다 |
| 새 기능 상태 | 아직 구현되지 않은 기능은 완료 기능처럼 표현하지 않는다 |

현재 데이터 구조:

| 필드 | 설명 |
|---|---|
| `ActionType` | `PasteText`, `LaunchFile`, `LaunchUrl`, `MediaAction` |
| `Content` | `PasteText`일 때 붙여넣을 문구 |
| `LaunchPath` | `LaunchFile`일 때 실행할 파일/폴더 경로 |
| `LaunchUrl` | `LaunchUrl`일 때 열 웹페이지 주소 |
| `MediaProvider` | `MediaAction`일 때 사용할 provider. 현재는 `System` |
| `MediaCommand` | `MediaAction`일 때 실행할 미디어 명령 |

현재 규칙:

- 기존 문구 데이터는 모두 `PasteText`로 유지된다.
- `LaunchFile`에서는 `Content`를 실행에 사용하지 않는다.
- `PasteText`에서는 `Content`가 필요하고, `LaunchPath`와 `LaunchUrl`은 비어 있을 수 있다.
- `LaunchFile`에서는 `LaunchPath`가 필요하고, `Content`와 `LaunchUrl`은 비어 있을 수 있다.
- `LaunchUrl`에서는 `LaunchUrl`이 필요하고, `Content`와 `LaunchPath`는 비어 있을 수 있다.
- `MediaAction`에서는 `MediaProvider`와 `MediaCommand`가 필요하고, `Content`, `LaunchPath`, `LaunchUrl`은 실행에 사용하지 않는다.
- 현재 `MediaProvider=System`은 Windows 범용 미디어 키 기반 제어를 의미한다.
- 실행 타입 확장은 문구 붙여넣기 안정성을 해치면 안 된다.

백업 ZIP 현재 구조:

| 항목 | 설명 |
|---|---|
| `launcher.db` | SQLite DB 스냅샷 |
| `images/` | 원본 이미지와 썸네일 |
| `icon-cache/` | 프로그램/파일 실행 카드의 자동 아이콘 캐시 |

백업 설정:

| 설정 | 설명 |
|---|---|
| `AutoBackupEnabled` | 변경 후 자동 백업을 사용할지 여부 |
| `BackupFolderPath` | 사용자가 지정한 외부 백업 폴더 경로 |
| `AutoBackupRetentionCount` | 유지할 최근 자동 백업 개수, 기본값 10 |
| `LastBackupCreatedAt` | 마지막 백업 생성 시각 |

백업 관련 유지 정책:

- 백업 폴더 경로는 앱 데이터 폴더 내부가 아니어야 한다.
- 백업 실패 기록은 로그에 남기되, 백업 ZIP 안에는 로그 파일을 포함하지 않는다.
- 창 위치 저장은 설정에 저장되지만 자동 백업 생성 트리거로 보지 않는다.

## 4. 다음 MVP: 데이터 안전성 보강

다음 MVP의 목표는 **데이터 안전성 보강**이다.

현재 앱은 백업 ZIP을 만들 수 있지만, 다음 기능은 아직 없다.

1. DB 오류 시 앱이 반복 크래시하지 않게 막는 최소 안전장치
2. 기존 백업 ZIP을 선택해 현재 앱 데이터로 복원하는 기능

이번 MVP에서는 새 실행 타입이나 새 UI 기능을 넓히지 않는다.  
사용자가 이미 만든 덱, 카드, 문구, 실행 경로, 웹 주소, 이미지 데이터를 잃지 않도록 안전장치를 보강하는 데 집중한다.

### 4.1 DB 오류 대응

기능 설명:

DB 오류 대응은 앱의 저장 파일이 잠기거나 손상되었을 때 앱이 시작하자마자 계속 꺼지는 상황을 줄이는 기능이다.

포함 범위:

- 앱이 저장하는 중 컴퓨터가 꺼져 DB 파일이 손상된 경우
- 백신, 동기화 프로그램, 다른 프로세스가 DB 파일을 잠깐 잡고 있는 경우
- DB 파일 경로가 이상하거나 접근 권한 문제가 생긴 경우

| ID | 요구사항 |
|---|---|
| FR-DB-001 | DB를 열거나 초기화하는 중 오류가 나도 앱이 반복 크래시하지 않아야 한다 |
| FR-DB-002 | DB 오류가 발생하면 사용자가 이해할 수 있는 상태 메시지 또는 안내를 보여준다 |
| FR-DB-003 | DB 오류 상세 내용은 로그에 남긴다 |
| FR-DB-004 | DB 오류 대응 중 기존 DB 파일을 임의로 삭제하지 않는다 |
| FR-DB-005 | 손상된 DB를 새 DB로 교체해야 하는 경우, 교체 전에 가능한 한 기존 DB 파일을 별도 백업한다 |
| FR-DB-006 | DB 오류 상태에서도 앱이 최소한 종료 안내 또는 복구 안내를 보여줄 수 있어야 한다 |

제외 범위:

- 복잡한 자동 복구
- 사용자 확인 없는 DB 삭제
- 백업 ZIP 복원과 섞인 대규모 구조 변경

성공 기준:

- DB 파일 잠금 또는 잘못된 DB 경로 상황에서 앱이 반복 크래시하지 않는다.
- DB 오류가 발생하면 사용자가 원인을 대략 이해할 수 있는 안내를 본다.
- 오류 내용이 로그에 남는다.
- 기존 DB 파일이 사용자 동의 없이 삭제되지 않는다.

주의사항:

- 사용자가 만든 데이터를 조용히 삭제하거나 덮어쓰면 안 된다.
- DB 오류 대응은 백업 복원 기능과 연결될 수 있지만, 두 기능은 분리해서 구현한다.

### 4.2 백업 ZIP 복원

기능 설명:

현재 앱은 백업 ZIP을 만들 수 있지만, 백업 ZIP을 앱 안에서 복원하는 기능은 없다.  
데이터 안전성 MVP에서는 사용자가 백업 ZIP을 선택해 현재 앱 데이터로 되돌릴 수 있게 한다.

포함 범위:

- SQLite DB 파일 `launcher.db`
- 사용자가 선택해 앱 내부에 저장된 원본 이미지
- 이미지 썸네일
- 실행 파일/바로 가기 자동 아이콘 캐시
- DB 안에 저장된 설정값, 실행 경로, 웹 주소

| ID | 요구사항 |
|---|---|
| FR-RESTORE-001 | 사용자는 설정 화면에서 백업 ZIP 복원을 시작할 수 있다 |
| FR-RESTORE-002 | 사용자는 기존 DeckDeckDeck 백업 ZIP 파일을 선택할 수 있다 |
| FR-RESTORE-003 | 앱은 선택한 ZIP 안에 필수 파일인 `launcher.db`가 있는지 확인한다 |
| FR-RESTORE-004 | 앱은 ZIP 안의 `images`, `icon-cache` 폴더가 있으면 함께 복원한다 |
| FR-RESTORE-005 | 복원 전에 현재 앱 데이터를 안전 백업으로 한 번 더 저장한다 |
| FR-RESTORE-006 | 복원은 현재 DB, 이미지, 썸네일, 자동 아이콘 캐시를 백업 ZIP 내용으로 교체한다 |
| FR-RESTORE-007 | 복원 실패 시 가능한 한 기존 현재 데이터가 유지되어야 한다 |
| FR-RESTORE-008 | 복원이 끝나면 앱 재시작이 필요하다는 안내를 보여준다 |
| FR-RESTORE-009 | 잘못된 ZIP, 필수 파일이 없는 ZIP, 읽을 수 없는 ZIP은 복원하지 않고 오류 메시지를 보여준다 |

제외 범위:

- 사용자가 카드에 연결한 외부 실행 파일 자체
- 사용자가 카드에 연결한 외부 문서 파일 자체
- 앱 로그 파일
- 빌드 결과물, 실행 파일, 개발용 임시 파일

성공 기준:

- 사용자가 백업 ZIP을 선택할 수 있다.
- 올바른 백업 ZIP을 선택하면 DB, 이미지, 썸네일, 자동 아이콘 캐시가 복원된다.
- 복원 전에 현재 데이터 안전 백업이 만들어진다.
- 잘못된 ZIP을 선택해도 앱이 종료되지 않는다.
- 복원 후 앱을 다시 열면 백업 시점의 덱과 카드가 표시된다.

주의사항:

- 복원은 현재 데이터를 덮어쓰는 위험한 작업이다.
- 복원 전 안전 백업이 실패하면 복원을 진행하지 않는 방향을 우선 검토한다.
- 복원 중 앱이 DB를 사용하고 있으면 파일 교체가 실패할 수 있으므로, 복원 흐름은 앱 재시작 또는 안전한 교체 절차를 고려해야 한다.
- 백업 ZIP이 다른 버전에서 만들어졌더라도 현재 앱이 읽을 수 있는지 확인해야 한다.

### 4.3 데이터 안전성 MVP 우선순위

1. DB 오류 대응
   - 확인 방법: DB 파일 잠금/손상/잘못된 DB 경로 상황에서도 앱이 반복 크래시하지 않는지 확인

2. 백업 ZIP 복원 기본 흐름
   - 확인 방법: 백업 ZIP을 선택하고 `launcher.db`, `images`, `icon-cache`를 앱 데이터로 되돌릴 수 있는지 확인

3. 복원 전 안전 백업
   - 확인 방법: 복원 직전에 현재 앱 데이터가 별도 ZIP으로 백업되는지 확인

4. 잘못된 ZIP 방어
   - 확인 방법: 필수 파일이 없는 ZIP, 깨진 ZIP, 읽을 수 없는 ZIP을 선택해도 앱이 종료되지 않는지 확인

5. 회귀 테스트
   - 확인 방법: 기존 문구 붙여넣기, 프로그램/파일 실행, 웹페이지 열기, 음악/미디어 제어, 자동 백업, 클립보드 복원, 자동 숨김이 그대로 작동하는지 확인

### 4.4 데이터 안전성 MVP 검증 체크리스트

- `dotnet test`가 통과한다.
- DB 오류 상황에서 앱이 반복 크래시하지 않는다.
- DB 오류가 발생하면 사용자 안내와 로그가 남는다.
- DB 오류 대응 중 기존 DB 파일이 조용히 삭제되지 않는다.
- 설정 화면에서 백업 ZIP 복원을 시작할 수 있다.
- 올바른 백업 ZIP을 선택하면 DB, 이미지, 썸네일, 자동 아이콘 캐시가 복원된다.
- 복원 전에 현재 데이터 안전 백업이 만들어진다.
- 잘못된 ZIP을 선택해도 앱이 종료되지 않는다.
- 복원 후 앱을 다시 열면 백업 시점의 덱과 카드가 표시된다.
- 기존 문구 카드가 그대로 붙여넣기 된다.
- 기존 프로그램/파일 실행 카드가 실행된다.
- 기존 웹페이지 주소 열기 카드가 기본 브라우저로 열린다.
- 기존 음악/미디어 제어 카드가 Windows 범용 미디어 키 명령을 실행한다.
- 기존 자동 백업이 계속 동작한다.
- 테스트 통과 후 최신 `DeckDeckDeck.exe`를 레포 루트에 준비한다.

### 5.1 음악/미디어 제어 카드

기능 설명:

새 카드 타입으로 **음악/미디어 제어**를 추가했다.
사용자-facing 명칭은 `음악/미디어 제어`로 한다. 내부 구현명은 `MediaAction` 정도로 둘 수 있다.

이 기능은 `멀티미디어 제어`와 `음악 제어`를 분리하지 않고 하나로 통합한다.
사용자는 Spotify, Apple Music, YouTube Music, 브라우저 재생을 구분하기보다 보통 `재생/일시정지`, `다음 곡`, `볼륨 증가`처럼 동작 중심으로 생각하기 때문이다.

1차 포함 범위:

- 재생/일시정지
- 이전 곡
- 다음 곡
- 정지
- 음소거
- 볼륨 증가
- 볼륨 감소
- Windows 범용 미디어 키 기반 제어
- Lucide 기반 기본 미디어 아이콘
- 실행 실패 시 상태 메시지와 로그 기록

| ID | 요구사항 |
|---|---|
| FR-MEDIA-001 | 사용자는 카드 타입으로 `음악/미디어 제어`를 선택할 수 있다 |
| FR-MEDIA-002 | 사용자는 재생/일시정지, 이전 곡, 다음 곡, 정지 중 하나를 명령으로 선택할 수 있다 |
| FR-MEDIA-003 | 사용자는 음소거, 볼륨 증가, 볼륨 감소 중 하나를 명령으로 선택할 수 있다 |
| FR-MEDIA-004 | 선택한 카드를 실행하면 Windows 범용 미디어 키 동작이 실행된다 |
| FR-MEDIA-005 | 미디어 제어 실행이 실패해도 앱이 종료되지 않고 상태 메시지와 로그를 남긴다 |
| FR-MEDIA-006 | 사용자가 직접 이미지를 지정하지 않은 미디어 카드에는 기본 미디어 아이콘을 표시한다 |

제외 범위:

- Spotify OAuth 로그인 및 API
- Apple Music API
- YouTube Music 전용 제어
- 현재 곡 좋아요
- 셔플
- 반복
- 특정 플레이리스트 재생
- 재생 장치 변경
- 현재 곡 정보 표시

성공 기준:

- 사용자는 미디어 명령 카드를 만들고 저장할 수 있다. 완료.
- 카드를 실행하면 Windows에서 현재 포커스 또는 기본 미디어 세션에 미디어 키 입력이 전달된다. 완료.
- Spotify, Apple Music, YouTube Music 같은 서비스 로그인이 없어도 1차 기능을 사용할 수 있다. 완료.
- 기존 문구 붙여넣기, 프로그램/파일 실행, 웹페이지 주소 열기 카드가 그대로 동작한다. 완료.

주의사항:

- 1차 범위는 서비스별 API가 아니라 Windows 범용 미디어 키 기반 제어로 제한한다.
- Spotify 같은 서비스 전용 제어는 같은 `MediaAction` 안의 provider 확장 MVP로 이어간다.
- 예시 구조는 다음과 같다.

```json
{
  "actionType": "media",
  "provider": "system",
  "command": "playPause"
}
```

향후 Spotify 확장 예시는 다음과 같다.

```json
{
  "actionType": "media",
  "provider": "spotify",
  "command": "likeCurrentTrack"
}
```

### 5.2 Spotify API provider MVP

기능 설명:

`음악/미디어 제어` 카드의 provider를 `System`에서 `Spotify`로 확장해 Spotify Web API 기반 조작을 지원한다.
새 카드 타입을 만들지 않고 기존 `MediaAction` 안에서 provider와 command를 확장한다.

Spotify API 연동은 로컬 앱 방향을 유지하되, 사용자가 명확히 켠 경우에만 OAuth 로그인을 진행한다.
데스크톱 앱에서는 클라이언트 시크릿을 안전하게 숨기기 어렵기 때문에 OAuth 2.0 Authorization Code with PKCE 흐름을 기본 후보로 둔다.

1차 포함 범위:

- Spotify 로그인 연결/해제
- 토큰 저장 및 갱신
- 재생/일시정지
- 이전 곡
- 다음 곡
- 현재 곡 좋아요
- 셔플 켜기/끄기
- 반복 모드 변경
- 현재 재생 상태 확인
- 사용 가능한 재생 장치 목록 확인
- 재생 장치 변경
- API 실패 시 상태 메시지와 로그 기록

후보 확장 범위:

- 특정 플레이리스트 재생
- 현재 곡 정보 표시
- 볼륨 설정
- 큐에 트랙 추가

| ID | 요구사항 |
|---|---|
| FR-SPOTIFY-001 | 사용자는 설정 화면에서 Spotify 연결을 시작할 수 있다 |
| FR-SPOTIFY-002 | Spotify 연결은 사용자가 명확히 선택한 경우에만 진행한다 |
| FR-SPOTIFY-003 | 데스크톱 앱 인증은 Authorization Code with PKCE 방식을 우선 사용한다 |
| FR-SPOTIFY-004 | 사용자는 `음악/미디어 제어` 카드에서 provider로 `Spotify`를 선택할 수 있다 |
| FR-SPOTIFY-005 | Spotify provider 명령은 기존 `System` provider 명령과 구분되어 저장된다 |
| FR-SPOTIFY-006 | Spotify API 호출 실패, 인증 만료, 재생 장치 없음 상황에서도 앱이 종료되지 않고 상태 메시지와 로그를 남긴다 |
| FR-SPOTIFY-007 | Spotify provider를 추가해도 Windows 범용 미디어 키 기반 `System` provider는 그대로 동작해야 한다 |

제외 범위:

- Apple Music API
- YouTube Music 전용 API
- 계정 기반 클라우드 동기화
- Spotify 계정 생성 또는 구독 관리
- Spotify 외 서비스 공통 OAuth 프레임워크

성공 기준:

- 사용자는 설정에서 Spotify 계정을 연결하고 해제할 수 있다.
- 사용자는 Spotify provider 미디어 카드를 만들고 저장할 수 있다.
- Spotify provider 카드 실행 시 Spotify Web API를 통해 선택한 명령이 실행된다.
- Spotify 인증 또는 API 호출이 실패해도 앱이 종료되지 않는다.
- 기존 Windows 범용 미디어 키 기반 미디어 카드와 기존 카드 타입이 그대로 동작한다.

주의사항:

- Spotify Web API는 사용자 권한 범위(scope)가 필요하므로, 필요한 권한만 요청해야 한다.
- 토큰은 비밀 정보로 취급하고 로그, 테스트 데이터, 예시값에 직접 남기지 않는다.
- Spotify API 호출은 네트워크와 계정 상태에 의존하므로 오프라인/만료/권한 거부 상태를 기본 실패 흐름으로 다뤄야 한다.
- 재생 장치가 없거나 Spotify 앱이 활성 장치를 만들지 않은 경우, 명령이 실패할 수 있다.
- Spotify API 정책 변경 가능성이 있으므로 구현 시점에는 공식 문서를 다시 확인한다.

구현 참고 문서:

- [Spotify Web API Authorization](https://developer.spotify.com/documentation/web-api/concepts/authorization)
- [Spotify Web API Player endpoints](https://developer.spotify.com/documentation/web-api/reference/get-information-about-the-users-current-playback)
- [Spotify Start/Resume Playback](https://developer.spotify.com/documentation/web-api/reference/start-a-users-playback)

### 5.3 Layout 편집

기능 설명:

Layout 편집은 기존 텐키 모양 15개 슬롯 구조 안에서 카드를 이동, 복사, 교환하는 기능이다.
자유 그리드 편집이 아니라 현재 슬롯 구조를 유지한 상태의 배치 편집으로 정의한다.

포함 범위:

- 카드 드래그 이동
- 빈 슬롯으로 카드 이동
- 카드 간 위치 교환
- 카드 복사
- 덱 슬롯 이동
- 우클릭 기반 복사/붙여넣기
- 변경 후 로컬 저장
- 실패 시 상태 메시지와 로그 기록

| ID | 요구사항 |
|---|---|
| FR-LAYOUT-001 | 사용자는 기존 15개 슬롯 안에서 카드를 다른 슬롯으로 이동할 수 있다 |
| FR-LAYOUT-002 | 사용자는 빈 슬롯으로 카드를 이동할 수 있다 |
| FR-LAYOUT-003 | 사용자는 카드 간 위치를 교환할 수 있다 |
| FR-LAYOUT-004 | 사용자는 카드를 복사해 다른 슬롯에 붙여넣을 수 있다 |
| FR-LAYOUT-005 | 사용자는 덱 슬롯 위치도 이동할 수 있다 |
| FR-LAYOUT-006 | 배치 변경은 로컬 DB에 저장된다 |
| FR-LAYOUT-007 | 저장 실패 시 앱이 종료되지 않고 상태 메시지와 로그를 남긴다 |

제외 범위:

- 복잡한 자유 배치
- 임의 크기 카드
- 커스텀 그리드 편집
- 텐키가 아닌 별도 레이아웃 체계

성공 기준:

- 사용자는 카드와 덱의 위치를 바꿀 수 있다.
- 앱을 다시 열어도 변경된 위치가 유지된다.
- 잘못된 저장 상황에서도 기존 데이터가 조용히 사라지지 않는다.

주의사항:

- Layout 편집은 15개 슬롯 구조를 흔들지 않는다.
- 이동, 복사, 교환은 사용자가 실수했을 때 이해하기 쉬운 피드백을 제공해야 한다.

### 5.4 Hotkey 실행 카드

기능 설명:

Hotkey 실행 카드는 사용자가 지정한 키 조합을 실행하는 카드 타입이다.
문구 붙여넣기나 파일 실행이 아니라, 앱 단축키를 빠르게 보내는 용도다.

포함 범위:

- 단일 키 또는 조합 키 입력
- `Ctrl`, `Alt`, `Shift`, `Win` 보조키 조합
- 실행 실패 시 상태 메시지와 로그 기록

| ID | 요구사항 |
|---|---|
| FR-HOTKEY-001 | 사용자는 카드 타입으로 `Hotkey 실행`을 선택할 수 있다 |
| FR-HOTKEY-002 | 사용자는 보조키와 일반 키를 조합해 저장할 수 있다 |
| FR-HOTKEY-003 | 카드를 실행하면 저장된 키 조합이 현재 활성 창으로 전달된다 |
| FR-HOTKEY-004 | 키 입력 실행 실패 시 앱이 종료되지 않고 상태 메시지와 로그를 남긴다 |

제외 범위:

- 명령어 실행과 인자 전달
- 앱별 조건 분기
- 여러 단계 자동화

성공 기준:

- 사용자는 자주 쓰는 앱 단축키를 카드로 실행할 수 있다.
- 기존 텐키 전역 단축키와 충돌하지 않도록 안내 또는 제한이 제공된다.

주의사항:

- 관리자 권한 앱이나 보호된 입력창에서는 키 전달이 제한될 수 있다.
- 이 기능은 문구 붙여넣기 안정성을 해치면 안 된다.

### 5.5 Multi Action + Delay

기능 설명:

Multi Action + Delay는 하나의 카드를 눌렀을 때 여러 동작을 순서대로 실행하고, 동작 사이에 대기 시간을 넣을 수 있는 자동화 기능이다.

포함 범위:

- 여러 카드 액션을 순서대로 실행
- 단계 사이 지연 시간 삽입
- 실행 중 실패한 단계 표시
- 실패 시 상태 메시지와 로그 기록

| ID | 요구사항 |
|---|---|
| FR-MULTI-001 | 사용자는 하나의 카드에 여러 실행 단계를 추가할 수 있다 |
| FR-MULTI-002 | 사용자는 단계 사이에 지연 시간을 설정할 수 있다 |
| FR-MULTI-003 | 실행 중 특정 단계가 실패하면 이후 진행 여부를 일관된 규칙으로 처리한다 |
| FR-MULTI-004 | 실패 내용은 상태 메시지와 로그에 남긴다 |

제외 범위:

- 조건문
- 반복문
- 복잡한 변수 시스템
- 외부 서비스 API 호출

성공 기준:

- 사용자는 간단한 반복 작업을 하나의 카드로 묶을 수 있다.
- 실패해도 앱이 종료되지 않는다.
- 사용자는 어느 단계에서 실패했는지 알 수 있다.

주의사항:

- 초기 범위는 단순 순차 실행과 지연 시간으로 제한한다.
- RPA 액션처럼 실패 가능성이 높은 기능과 섞을 때는 실험 기능 표시가 필요하다.

### 5.6 앱별 덱 / Smart Profiles

기능 설명:

사용자-facing 명칭은 **앱별 덱**으로 한다. 내부 개념으로 `Smart Profiles`를 사용할 수 있다.
활성 앱에 따라 기본으로 열리는 덱을 자동 전환한다.

예시:

- VS Code가 활성 앱이면 개발 덱
- Chrome이 활성 앱이면 브라우저 덱
- Discord가 활성 앱이면 음성채팅 덱
- OBS Studio가 활성 앱이면 방송 덱

초기 포함 범위:

- 앱별 덱 자동 전환 On/Off
- 프로세스 이름 기준 매칭
- 프로세스 이름과 덱 매핑
- 수동으로 현재 덱 고정
- 편집 중에는 자동 전환 비활성화

| ID | 요구사항 |
|---|---|
| FR-SMART-001 | 사용자는 앱별 덱 자동 전환을 켜거나 끌 수 있다 |
| FR-SMART-002 | 사용자는 프로세스 이름과 덱을 매핑할 수 있다 |
| FR-SMART-003 | 활성 앱 프로세스가 매핑과 일치하면 기본으로 열리는 덱이 자동 전환된다 |
| FR-SMART-004 | 사용자는 현재 덱을 수동으로 고정할 수 있다 |
| FR-SMART-005 | 카드 또는 덱 편집 중에는 자동 전환이 동작하지 않는다 |

제외 또는 후순위:

- 창 제목 조건
- 웹사이트 URL별 덱 전환
- 프로젝트별 덱 전환
- 복잡한 우선순위 규칙
- 클라우드 동기화

성공 기준:

- 사용자는 앱별로 자주 쓰는 덱을 연결할 수 있다.
- 활성 앱이 바뀌면 지정된 덱이 기본 덱으로 열린다.
- 편집 중 자동 전환 때문에 사용자가 수정 중인 화면을 잃지 않는다.

주의사항:

- 초기 매칭은 프로세스 이름 기준으로 제한한다.
- 자동 전환보다 사용자의 수동 고정이 우선해야 한다.
- 로컬 앱 방향을 유지하며 계정 기반 동기화는 포함하지 않는다.

## 6. 실험 기능 후보

### 6.1 화면 버튼 클릭 / UI 자동화 / RPA 액션

기능 설명:

사용자가 특정 앱이나 웹 화면의 버튼을 자동으로 누르고 싶어 할 수 있다.
예를 들어 ChatGPT 웹에서 `더 많은 액션` 버튼을 클릭한 뒤 `새 채팅에서 브랜치`를 클릭하는 흐름이다.

이 기능은 유용하지만 실패 가능성이 높으므로 핵심 MVP가 아니라 **실험 기능** 또는 **고급 자동화 후보**로 둔다.

권장 포함 범위:

- 현재 활성 창 기준 클릭
- 좌표 기반 클릭
- 지연 시간 삽입
- 여러 단계 클릭
- 실패 시 상태 메시지와 로그 기록

| ID | 요구사항 |
|---|---|
| FR-RPA-001 | 사용자는 실험 기능으로 화면 버튼 클릭 액션을 만들 수 있다 |
| FR-RPA-002 | 사용자는 현재 활성 창 기준 좌표 클릭을 저장할 수 있다 |
| FR-RPA-003 | 사용자는 여러 클릭 단계와 지연 시간을 순서대로 저장할 수 있다 |
| FR-RPA-004 | 클릭 실행 실패 시 앱이 종료되지 않고 상태 메시지와 로그를 남긴다 |

향후 확장 후보:

- UI Automation 기반 버튼 이름 검색
- 프로세스 이름 조건
- 창 제목 조건
- 이미지 매칭 기반 클릭
- 브라우저 확장 프로그램 연동

성공 기준:

- 사용자는 고급/실험 기능임을 인지한 상태에서 간단한 클릭 자동화를 만들 수 있다.
- 실패해도 앱이 종료되지 않는다.
- 실패 원인을 추적할 수 있도록 로그가 남는다.

주의사항:

- 웹 UI는 자주 바뀌므로 안정성이 낮을 수 있다.
- 브라우저 내부 버튼은 Windows UI Automation으로 항상 안정적으로 찾을 수 없다.
- 이 기능은 기본 사용자 기능이 아니라 고급/실험 기능으로 둔다.
- 실패해도 앱이 종료되지 않아야 한다.

## 7. 장기 확장

### 7.1 서비스 provider 일반화

기능 설명:

Spotify API provider MVP가 안정화된 뒤, 같은 구조로 다른 서비스 전용 provider를 검토할 수 있다.
이때도 별도 카드 타입을 늘리기보다 같은 `MediaAction` 안에서 provider를 확장한다.

후보 범위:

- Spotify provider에서 검증한 provider 인터페이스 일반화
- 다른 음악/미디어 서비스 provider 추가
- provider별 권한 안내
- provider별 실패 메시지 표준화
- provider별 인증 저장/해제 흐름 표준화

| ID | 요구사항 |
|---|---|
| FR-PROVIDER-001 | 서비스 전용 제어는 기존 `음악/미디어 제어` 카드의 provider 확장으로 정의한다 |
| FR-PROVIDER-002 | provider 확장은 기본 system provider 기능을 깨뜨리지 않아야 한다 |
| FR-PROVIDER-003 | 외부 서비스 인증이 필요한 기능은 사용자가 명확히 켜는 선택 기능으로 둔다 |

제외 범위:

- 데이터 안전성 MVP
- 음악/미디어 제어 1차 범위
- Spotify API provider MVP
- 계정 기반 클라우드 동기화

주의사항:

- 외부 서비스 provider 일반화는 유지보수 비용과 인증 리스크가 크므로 Spotify provider 안정화 뒤 검토한다.
- 로컬 앱 방향을 해치지 않도록 선택 기능으로 제한한다.

### 7.2 브라우저 확장 연동

기능 설명:

브라우저 내부 버튼이나 웹페이지 상태를 더 안정적으로 다루기 위해 장기적으로 브라우저 확장 프로그램 연동을 검토할 수 있다.

후보 범위:

- 현재 탭 정보 읽기
- 특정 웹앱 버튼 실행
- 웹사이트 URL별 덱 전환 보조
- RPA 액션 안정성 보강

| ID | 요구사항 |
|---|---|
| FR-BROWSER-001 | 브라우저 확장 연동은 Windows 앱 단독 기능이 충분히 검증된 뒤 검토한다 |
| FR-BROWSER-002 | 브라우저 확장은 사용자가 명확히 설치하고 켜는 선택 기능이어야 한다 |

제외 범위:

- 초기 RPA 실험 기능
- 데이터 안전성 MVP
- 기본 웹페이지 주소 열기 카드

주의사항:

- 브라우저별 구현 차이가 크므로 초기 MVP에 포함하지 않는다.
- 확장 프로그램 설치가 필요한 기능은 개인용 로컬 실행 정책과 충돌하지 않는지 별도 검토한다.

### 7.3 플러그인 구조

기능 설명:

카드 타입과 외부 연동이 충분히 늘어난 뒤, 장기적으로 플러그인 구조를 검토할 수 있다.

후보 범위:

- 카드 타입 확장 지점
- provider 확장 지점
- 고급 자동화 액션 확장 지점
- 플러그인별 권한 안내

| ID | 요구사항 |
|---|---|
| FR-PLUGIN-001 | 플러그인 구조는 핵심 카드 타입이 안정화된 뒤 검토한다 |
| FR-PLUGIN-002 | 플러그인은 로컬 데이터 안전성과 실행 안정성을 해치지 않아야 한다 |
| FR-PLUGIN-003 | 플러그인이 외부 서비스 또는 파일 접근을 요구하면 사용자에게 명확히 안내해야 한다 |

제외 범위:

- 데이터 안전성 MVP
- 음악/미디어 제어 1차 범위
- Layout 편집
- 앱별 덱 초기 범위

주의사항:

- 플러그인 구조는 제품 복잡도를 크게 올리므로 장기 확장으로 둔다.
- 몇 개의 내장 카드 타입으로 충분한 동안은 플러그인보다 내장 기능 안정화가 우선이다.
