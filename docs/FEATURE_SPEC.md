# FEATURE_SPEC — 전역 핫키 직접 실행

## 1. 기능 개요

DeckDeckDeck의 기존 실행 흐름은 전역 단축키로 홈 또는 카테고리를 연 뒤, 텐키 슬롯을 한 번 더 선택하는 2단계 방식이다.

이번 변경에서는 사용자가 별도의 전역 핫키와 실행 동작을 직접 등록하고, 등록한 키를 한 번 누르는 것만으로 해당 동작을 즉시 실행할 수 있어야 한다.

예시:

- `Ctrl + Shift + K` → 볼륨 증가
- `오른쪽 화살표` → 다음 곡

새 핫키 기능은 기존 슬롯의 바로 가기가 아니다. 각 핫키는 핫키 설정 화면에서 독립적인 실행 항목으로 생성하고 관리한다.

---

## 2. 해결하려는 사용자 문제

현재 사용자는 자주 사용하는 동작을 실행하기 위해 다음 과정을 거쳐야 한다.

1. `Ctrl + 텐키 키`를 눌러 홈 또는 카테고리를 연다.
2. 실행할 슬롯의 텐키 키를 다시 누른다.

볼륨 조절, 다음 곡, 정해진 문구 붙여넣기처럼 반복적으로 사용하는 동작에는 이 과정이 불필요하게 길다.

사용자는 자주 쓰는 동작에 원하는 키를 지정하고, DeckDeckDeck 화면을 거치지 않은 채 즉시 실행할 수 있어야 한다.

---

## 3. 범위

### 포함

- 홈 그리드의 `Hotkey` 영역을 통한 핫키 관리 화면 진입
- 여러 개의 핫키 전용 실행 항목 생성, 조회, 편집 및 삭제
- 단일 키 또는 조합 키 등록
- 보조 키가 없는 일반 키의 전역 등록
- 현재 실행 항목이 지원하는 모든 실행 타입 지원
- 등록된 키의 원래 입력을 차단하고 설정된 동작으로 대체
- 핫키를 누르고 있는 동안 한 번만 실행
- 저장 직후 핫키 활성화
- 기존 백업 및 복원 흐름과의 연동

### 포함하지 않음

- 기존 `Ctrl + Numpad` 홈·카테고리 진입 방식의 변경 또는 제거
- 기존 카테고리 및 슬롯 구조의 변경
- 기존 슬롯에 핫키를 연결하는 기능
- 새로운 실행 타입 추가
- 모바일 환경 지원
- 핫키별 키 반복 실행 설정
- 핫키 실행을 위한 별도의 알림 또는 토스트 체계 추가
- 핫키 충돌 정책의 별도 정의

## 4. 변경 전후 사용자 흐름

### 변경 전

1. 사용자가 다른 프로그램을 사용한다.
2. `Ctrl + 텐키 키`를 누른다.
3. DeckDeckDeck 홈 또는 카테고리 팔레트가 표시된다.
4. 사용자가 실행할 슬롯의 텐키 키를 누른다.
5. 해당 슬롯의 실행 항목이 실행된다.

### 변경 후 — 기존 흐름

기존 흐름은 그대로 유지한다.

1. 사용자가 `Ctrl + 텐키 키`를 누른다.
2. 홈 또는 카테고리 팔레트가 표시된다.
3. 사용자가 텐키 슬롯을 선택한다.
4. 기존 실행 항목이 실행된다.

### 변경 후 — 직접 실행 핫키

1. 사용자가 다른 프로그램을 사용한다.
2. 등록된 전역 핫키를 한 번 누른다.
3. 해당 키의 원래 입력은 대상 프로그램에 전달되지 않는다.
4. DeckDeckDeck 홈, 카테고리 또는 팔레트는 표시되지 않는다.
5. 핫키에 저장된 실행 동작이 즉시 실행된다.
6. 실행 결과와 오류 처리는 동일한 실행 타입의 기존 슬롯 실행 동작을 따른다.

## 5. 핫키 관리 화면 진입

### 홈 화면

- 홈 그리드 왼쪽 위의 기존 `Hotkey` 더미 영역을 실제 버튼으로 제공한다.
- 사용자가 이 영역을 클릭하거나 키보드로 활성화하면 핫키 관리 화면으로 이동한다.
- 핫키 관리 화면에서 뒤로 가면 홈 화면으로 돌아간다.

### 카테고리 화면

- 카테고리 그리드에도 `Hotkey` 영역은 현재 위치에 계속 표시한다.
- 카테고리 화면의 `Hotkey` 영역은 비활성 상태여야 한다.
- 비활성 영역은 마우스 클릭과 키보드 포커스를 받지 않아야 한다.
- 비활성 상태임을 시각적으로 구분할 수 있어야 한다.
- 카테고리 화면에서는 핫키 관리 화면으로 진입할 수 없다.

---

## 6. 핫키 관리 화면

핫키 관리 화면은 저장된 핫키 전용 실행 항목의 목록을 표시한다.

각 항목에서는 최소한 다음 정보를 확인할 수 있어야 한다.

- 실행 항목 이름
- 등록된 키 또는 키 조합
- 실행 타입
- 실행 동작의 요약
- 사용 여부

사용자는 다음 행동을 할 수 있어야 한다.

- 새 핫키 추가
- 기존 핫키 편집
- 기존 핫키 삭제
- 핫키 사용 또는 사용 중지
- 홈으로 돌아가기

### 빈 상태

등록된 핫키가 없으면 다음 내용을 표시한다.

- 등록된 핫키가 없다는 안내
- 핫키를 추가하면 한 번의 키 입력으로 동작을 실행할 수 있다는 설명
- `핫키 추가` 행동

빈 상태에서도 사용자는 홈으로 돌아갈 수 있어야 한다.

---

## 7. 핫키 생성 및 편집

새 핫키 생성과 기존 핫키 편집은 동일한 입력 구조를 사용한다.

### 기본 정보

사용자는 다음 값을 설정할 수 있어야 한다.

- 실행 항목 이름
- 전역 핫키
- 실행 타입
- 실행 타입별 상세 설정
- 설명
- 이 핫키 사용 여부
- 이미지

### 실행 타입

핫키 전용 실행 항목은 기존 슬롯과 동일한 실행 타입을 지원한다.

- 문구 붙여넣기
- 파일, 폴더 또는 바로 가기 실행
- 웹 주소 열기
- 음악·미디어 제어
- 터미널 명령 실행

각 실행 타입에서 표시되는 입력 항목, 선택 가능한 동작, 입력 검증 및 오류 문구는 기존 실행 항목 편집 경험과 동일해야 한다.

예시:

- 문구 붙여넣기는 붙여넣을 문구와 붙여넣기 방식을 설정한다.
- 파일 실행은 실행할 파일, 폴더 또는 바로 가기를 지정한다.
- 웹 주소 열기는 실행할 주소를 지정한다.
- 음악·미디어 제어는 제어 대상과 명령을 선택한다.
- 터미널 명령은 셸, 관리자 권한 여부 및 명령을 설정한다.

### 저장

- 필수 입력이 유효한 경우에만 저장할 수 있어야 한다.
- 저장에 성공하면 핫키 관리 목록으로 돌아간다.
- 저장된 핫키는 앱을 다시 시작하지 않아도 즉시 사용할 수 있어야 한다.
- 새 항목을 저장하면 목록에 추가되어야 한다.
- 기존 항목을 저장하면 목록의 내용이 갱신되어야 한다.
- 저장 결과는 기존 실행 항목 및 설정 저장 UX와 동일한 상태 메시지 방식을 사용한다.
- 저장 실패 시 편집 중인 입력값은 유지되어야 한다.
- 저장 실패 시 사용자가 원인을 확인하고 다시 저장할 수 있어야 한다.

### 뒤로 가기 및 취소

- 저장하지 않고 뒤로 가면 편집 내용을 적용하지 않는다.
- 뒤로 가기 동작은 기존 실행 항목 편집 화면의 취소 경험을 따른다.
- 취소 후에는 핫키 관리 목록으로 돌아간다.

### 삭제

- 저장된 핫키 전용 실행 항목만 삭제할 수 있다.
- 삭제 전에는 확인을 요청해야 한다.
- 사용자가 삭제를 취소하면 항목과 설정은 유지되어야 한다.
- 삭제가 완료되면 해당 키는 더 이상 동작을 실행하지 않아야 한다.
- 삭제 후 핫키 관리 목록으로 돌아간다.

## 8. 핫키 입력 UX

### 입력 시작

- 사용자가 핫키 입력 영역을 활성화하면 다음 키 입력을 기다리는 상태가 되어야 한다.
- 대기 중임을 시각적으로 명확히 표시해야 한다.
- 핫키 입력 대기 중에는 이미 등록된 직접 실행 핫키가 실행되지 않아야 한다.
- 사용자가 입력한 키는 설정값으로만 사용되며 다른 프로그램이나 현재 화면의 일반 입력으로 전달되지 않아야 한다.

### 허용되는 입력

다음 형태를 등록할 수 있어야 한다.

- 보조 키가 포함된 조합
  - 예: `Ctrl + Shift + K`
- 보조 키가 없는 단일 키
  - 예: `오른쪽 화살표`
- 텐키 및 기능 키를 포함한 일반적인 물리 키

`Ctrl`, `Shift`, `Alt`, `Win`과 같은 보조 키만 단독으로 누른 상태는 완성된 핫키로 저장하지 않는다. 보조 키가 없는 단일 키는 허용한다.

### 표시

- 현재 등록된 조합을 사람이 이해할 수 있는 형식으로 표시한다.
- 보조 키와 본 키의 구분이 명확해야 한다.
- 예: `Ctrl + Shift + K`, `Right Arrow`, `Numpad 7`
- 키를 변경하기 전에는 기존 값이 유지되어야 한다.
- 사용자가 명시적으로 입력을 완료했을 때만 새 값으로 교체한다.

### 입력 취소 및 초기화

- 핫키 입력 대기 중 `Esc`를 누르면 새 입력을 취소하고 기존 값을 유지한다.
- 별도의 초기화 행동을 통해 등록된 키를 지울 수 있어야 한다.
- 키가 비어 있는 항목은 활성 상태로 저장할 수 없어야 한다.

## 9. 직접 실행 동작

### 전역 동작

- 앱이 실행 중이면 DeckDeckDeck 창의 표시 여부와 관계없이 등록된 핫키를 감지해야 한다.
- 사용자가 다른 프로그램을 사용 중이어도 핫키가 동작해야 한다.
- 보조 키가 없는 단일 키도 동일하게 전역에서 동작해야 한다.
- 비활성화된 핫키는 입력을 감지하거나 원래 키 입력을 차단하지 않아야 한다.

### 원래 키 입력 대체

활성화된 핫키가 감지되면 해당 키의 원래 입력은 다른 프로그램에 전달되지 않아야 한다.

예시:

- `오른쪽 화살표`가 `다음 곡`으로 등록된 경우:
  - 다음 곡 동작을 실행한다.
  - 문서의 커서 이동, 브라우저 이동 등 원래 오른쪽 화살표 동작은 발생하지 않는다.

### 한 번만 실행

- 키를 처음 누른 시점에 동작을 한 번 실행한다.
- 키를 계속 누르고 있어도 같은 동작을 반복 실행하지 않는다.
- 키를 완전히 뗀 뒤 다시 눌러야 다음 실행이 가능하다.
- 이 규칙은 볼륨 증가·감소를 포함한 모든 실행 타입에 동일하게 적용된다.

### 창 표시

- 직접 실행 핫키를 눌러도 홈, 카테고리 또는 실행 팔레트를 표시하지 않는다.
- 핫키 실행을 위해 DeckDeckDeck 창을 앞으로 가져오지 않는다.
- 동작 자체가 파일, 웹페이지, Spotify 또는 터미널 창을 여는 경우에는 해당 실행 결과로 발생하는 창 표시를 허용한다.

### 실행 상태와 피드백

- 별도의 성공 토스트나 신규 알림 UI를 추가하지 않는다.
- 성공, 실패 및 상태 메시지는 기존 슬롯 실행에서 사용하는 방식과 문구를 따른다.
- 동작 자체가 실패하더라도 원래 키 입력을 뒤늦게 전달하지 않는다.
- 실행 도중 오류가 발생해도 앱이 종료되거나 이후 핫키 입력이 불가능해져서는 안 된다.

## 10. 실행 타입별 기존 동작 유지

### 문구 붙여넣기

- 핫키를 누르는 시점에 사용자가 작업 중인 창을 붙여넣기 대상으로 사용한다.
- 핫키 자체의 키 입력은 붙여넣기 대상에 전달하지 않는다.
- 기존의 붙여넣기 방식 설정을 동일하게 적용한다.
- 기존의 클립보드 복원 설정을 동일하게 적용한다.
- 붙여넣기 실패 시 기존 슬롯 실행과 동일한 오류 처리를 적용한다.

### 파일, 폴더 및 바로 가기 실행

- 저장된 대상을 기존 슬롯 실행과 동일하게 연다.
- 대상이 없거나 실행할 수 없으면 기존 오류 처리를 따른다.

### 웹 주소 열기

- 저장된 주소를 기존 슬롯 실행과 동일하게 연다.
- 주소 입력과 정규화 규칙은 기존 실행 항목과 동일하다.

### 음악·미디어 제어

- Windows 기본 미디어 제어와 Spotify 제어를 기존 실행 항목과 동일하게 지원한다.
- Spotify 연결이 필요한 동작은 기존 연결 상태와 오류 처리를 따른다.
- `볼륨 증가`, `볼륨 감소`, `다음 곡` 등의 명령도 길게 누르기에 의해 반복 실행되지 않는다.

### 터미널 명령 실행

- 저장된 셸, 관리자 권한 여부 및 명령을 기존 슬롯과 동일하게 적용한다.
- 관리자 권한 실행 시 기존 Windows 확인 흐름을 유지한다.
- 기존 안전 안내와 입력 검증을 유지한다.

## 11. 상태별 동작

### 초기 로딩

- 저장된 핫키 목록을 불러오는 동안 화면 구조가 깨지거나 이전 목록이 잘못 표시되어서는 안 된다.
- 로딩이 완료되면 현재 저장 상태를 기준으로 목록을 표시한다.

### 빈 상태

- 등록된 핫키가 없음을 명확히 표시한다.
- 사용자가 바로 새 핫키를 추가할 수 있어야 한다.

### 편집 중

- 사용자가 입력한 값은 저장하거나 취소할 때까지 화면에 유지되어야 한다.
- 실행 타입을 변경하면 해당 타입에 필요한 입력 영역을 표시한다.
- 실행 타입을 변경해도 저장하기 전까지 기존 저장 항목에는 영향을 주지 않는다.

### 저장 중

- 동일한 저장 요청을 연속으로 제출할 수 없어야 한다.
- 저장이 끝나기 전에는 화면을 중복 전환하지 않아야 한다.

### 저장 성공

- 목록 화면으로 돌아간다.
- 최신 이름, 키 조합, 실행 타입과 상태를 표시한다.
- 저장한 핫키는 즉시 사용할 수 있어야 한다.

### 저장 실패

- 편집 화면에 머무른다.
- 사용자가 입력한 값은 유지한다.
- 오류 내용을 확인할 수 있어야 한다.
- 사용자가 값을 수정하거나 동일한 내용으로 다시 저장할 수 있어야 한다.

### 실행 중

- 같은 키를 계속 누르고 있어도 추가 실행하지 않는다.
- 실행이 완료되거나 실패한 뒤 키를 뗐다가 다시 누르면 다시 실행할 수 있다.

### 실행 실패

- 기존 실행 타입의 오류 처리와 상태 메시지를 사용한다.
- DeckDeckDeck 팔레트를 자동으로 열지 않는다.
- 이후 다른 핫키를 계속 사용할 수 있어야 한다.

## 12. 기존 기능과의 관계

### 기존 전역 단축키

- `Ctrl + Numpad 0` 홈 열기 동작을 유지한다.
- `Ctrl + Numpad 0` 길게 누르기 최소화 동작을 유지한다.
- `Ctrl + Numpad 1~9`, `/`, `*`, `-`, `+`, `.` 카테고리 바로 열기 동작을 유지한다.
- 팔레트가 열린 뒤 텐키로 슬롯을 선택하는 동작을 유지한다.

### 기존 슬롯

- 카테고리와 슬롯의 생성, 편집, 삭제, 복사, 이동 및 활성화 기능을 변경하지 않는다.
- 핫키 전용 실행 항목은 기존 슬롯을 참조하지 않는다.
- 기존 슬롯을 수정하거나 삭제해도 핫키 전용 실행 항목은 변경되지 않는다.
- 핫키 전용 실행 항목을 수정하거나 삭제해도 기존 슬롯은 변경되지 않는다.

### 일반 설정

- 창 표시, 붙여넣기 후 숨기기, 클립보드 복원, Spotify 연결 등 기존 설정을 유지한다.
- 핫키 전용 실행 항목은 실행 시 관련된 기존 설정을 동일하게 적용한다.

### 백업 및 복원

- 핫키 전용 실행 항목도 기존 사용자 데이터의 일부로 취급한다.
- 기존 수동 백업, 자동 백업 및 ZIP 복원에 핫키 설정이 함께 포함되어야 한다.
- 복원 후 앱 재시작이 필요한 기존 UX는 그대로 유지한다.
- 복원된 핫키는 재시작 후 저장 당시의 키, 실행 동작, 설명, 이미지 및 사용 상태를 유지해야 한다.

## 13. 데스크톱 및 접근성 요구사항

### 데스크톱

- 이 기능은 현재 지원 대상인 Windows 데스크톱 환경을 기준으로 한다.
- 창이 숨겨졌거나 최소화된 상태에서도 활성화된 핫키가 동작해야 한다.
- 다중 모니터 사용 여부와 관계없이 직접 실행을 위해 앱 창을 이동하거나 표시하지 않는다.

### 키보드 접근성

- 홈의 `Hotkey` 버튼은 키보드 포커스를 받을 수 있어야 한다.
- `Enter` 또는 `Space`로 핫키 관리 화면에 진입할 수 있어야 한다.
- 핫키 목록, 추가, 편집, 삭제, 저장, 취소 및 초기화 행동을 마우스 없이 사용할 수 있어야 한다.
- 비활성화된 카테고리의 `Hotkey` 영역은 탭 순서에 포함되지 않아야 한다.

### 화면 읽기 및 상태 전달

- 핫키 입력 영역은 현재 값과 입력 대기 상태를 구분해 전달해야 한다.
- 각 목록 항목은 이름, 키 조합, 실행 타입 및 사용 상태를 식별할 수 있어야 한다.
- 저장 오류는 포커스 이동 없이도 인식할 수 있어야 한다.
- 포커스 표시 방식은 기존 앱의 포커스 스타일과 일관되어야 한다.

## 14. 주요 엣지케이스

- 핫키 입력 대기 중 기존 직접 실행 핫키를 눌러도 동작이 실행되지 않고 새 입력으로만 처리되어야 한다.
- 비활성화된 핫키의 키를 누르면 원래 키 입력이 정상적으로 대상 프로그램에 전달되어야 한다.
- 활성화된 단일 키 핫키는 DeckDeckDeck이 화면에 보이지 않아도 원래 입력을 차단해야 한다.
- 키를 누른 채 자동 반복이 발생해도 동작은 한 번만 실행되어야 한다.
- 실행이 실패해도 해당 키를 뗀 뒤 다시 누르면 재시도할 수 있어야 한다.
- 편집 중인 항목을 저장하지 않고 나가면 기존 저장 값이 바뀌지 않아야 한다.
- 핫키 전용 실행 항목을 삭제해도 같은 동작을 가진 기존 슬롯은 영향을 받지 않아야 한다.
- 기존 슬롯을 삭제하거나 이동해도 독립적인 핫키 전용 실행 항목은 영향을 받지 않아야 한다.
- Spotify가 연결되지 않은 상태에서도 Spotify 실행 항목을 저장할 수 있으며, 실행 시 기존 Spotify 미연결 UX를 따라야 한다.
- 앱을 종료한 상태에서는 핫키가 입력을 차단하거나 동작을 실행하지 않아야 한다.

## 15. 완료 검증 기준

### 진입 및 화면

- [ ] 홈 그리드의 `Hotkey` 영역을 클릭하면 핫키 관리 화면이 열린다.
- [ ] 홈의 `Hotkey` 영역을 키보드로 활성화할 수 있다.
- [ ] 카테고리 화면의 `Hotkey` 영역은 표시되지만 클릭하거나 포커스할 수 없다.
- [ ] 핫키 관리 화면에서 뒤로 가면 홈으로 돌아간다.
- [ ] 등록 항목이 없을 때 빈 상태와 `핫키 추가` 행동이 표시된다.

### 생성 및 편집

- [ ] 핫키 설정 화면에서 기존 슬롯과 독립된 실행 항목을 만들 수 있다.
- [ ] 모든 기존 실행 타입을 선택하고 동일한 상세 설정을 입력할 수 있다.
- [ ] `Ctrl + Shift + K`와 같은 조합 키를 등록할 수 있다.
- [ ] `오른쪽 화살표`와 같은 보조 키 없는 단일 키를 등록할 수 있다.
- [ ] 보조 키만으로는 활성 핫키를 저장할 수 없다.
- [ ] 저장하지 않고 뒤로 가면 변경 내용이 반영되지 않는다.
- [ ] 저장 실패 시 입력값이 유지되고 다시 저장할 수 있다.
- [ ] 저장된 항목을 편집, 비활성화 및 삭제할 수 있다.
- [ ] 삭제 전 확인이 표시된다.

### 직접 실행

- [ ] 저장 직후 앱 재시작 없이 새 핫키가 동작한다.
- [ ] 다른 프로그램 사용 중에도 활성 핫키가 동작한다.
- [ ] 직접 실행 시 DeckDeckDeck 팔레트가 표시되지 않는다.
- [ ] 활성 핫키의 원래 키 입력은 대상 프로그램에 전달되지 않는다.
- [ ] 비활성 핫키의 키 입력은 대상 프로그램에 정상적으로 전달된다.
- [ ] 키를 누르고 있어도 동작은 한 번만 실행된다.
- [ ] 키를 뗀 뒤 다시 누르면 다시 실행된다.
- [ ] 실행 실패 후에도 다른 핫키와 재입력이 정상 동작한다.

### 실행 타입

- [ ] 문구 붙여넣기는 핫키 입력 시점의 작업 창을 대상으로 실행된다.
- [ ] 파일, 폴더 및 바로 가기 실행이 기존 슬롯과 동일하게 동작한다.
- [ ] 웹 주소 열기가 기존 슬롯과 동일하게 동작한다.
- [ ] Windows 및 Spotify 미디어 제어가 기존 슬롯과 동일하게 동작한다.
- [ ] 터미널 명령 실행과 관리자 권한 흐름이 기존 슬롯과 동일하게 동작한다.
- [ ] 기존 클립보드 복원 및 Spotify 연결 설정이 직접 실행에도 적용된다.

### 기존 기능 보존

- [ ] 기존 `Ctrl + Numpad` 홈 및 카테고리 진입이 계속 동작한다.
- [ ] 기존 팔레트의 두 번째 텐키 선택이 계속 동작한다.
- [ ] 기존 슬롯을 수정하거나 삭제해도 핫키 전용 실행 항목은 변경되지 않는다.
- [ ] 핫키 전용 실행 항목을 수정하거나 삭제해도 기존 슬롯은 변경되지 않는다.
- [ ] 기존 백업 및 복원에 핫키 전용 실행 항목이 포함된다.
